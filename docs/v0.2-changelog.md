# Stitch Design Skill v0.2 — Requirements

> Выводы из первого тестового запуска скилла v0.1.0 (2026-03-19).
> Контекст: генерация UI для менторинг-сайта с календарём бронирования.

---

## Проблемы v0.1

- MCP `generate_screen_from_text` падает с `ECONNRESET` через ~60 сек (долгая операция, сервер разрывает соединение)
- Скилл не содержит fallback-стратегии → агент хаотично перепробовал 7 подходов за 15+ tool-вызовов
- Создано 5 дублирующих проектов в Stitch вместо 1
- Генерации на самом деле отработали на сервере, но агент узнал об этом случайно через `list_screens`
- context-map.md и usage.json так и не были записаны
- Промпт для Stitch показан на английском, хотя пользователь пишет по-русски
- Слишком много вопросов перед первой генерацией

---

## Требования к v0.2

### Управление проектами

**R-01. Один проект на дизайн-задачу.**
Все экраны, варианты и правки в рамках одной задачи — в одном проекте Stitch. Запретить создание нового проекта при retry или повторной генерации.

**R-02. Проверка существующих проектов перед созданием.**
Перед `create_project` обязательно вызвать `list_projects` и проверить `context-map.md`. Если есть активный проект для текущей задачи — переиспользовать его.

**R-03. Немедленная запись project ID в context-map.md.**
Сразу после `create_project` (до генерации) записать ID в `context-map.md`. Это гарантирует, что при сбое генерации проект не потеряется и не будет создан дубликат.

### Обработка ошибок и retry

**R-04. Fire-and-poll стратегия для долгих операций.**
Для `generate_screen_from_text`, `edit_screens`, `generate_variants`:
1. Отправить запрос
2. Если получен ECONNRESET / fetch failed / timeout — НЕ считать это провалом
3. Подождать 90 секунд
4. Опросить `list_screens` (быстрый вызов) — проверить, появился ли новый экран
5. Если найден → скачать артефакты, продолжить нормально
6. Если не найден → повторить poll через 60 секунд
7. Если после 2 poll'ов не найден → сообщить пользователю, предложить retry

**R-05. Запрет на создание нового проекта при retry.**
При повторной попытке генерации всегда использовать тот же project ID. Явно указать это в скилле как правило.

**R-06. Health-check перед генерацией.**
Перед долгой операцией выполнить быстрый вызов (`list_projects` или `list_screens`) чтобы убедиться, что MCP-соединение работает. Если быстрый вызов падает — предупредить пользователя о проблеме с подключением до начала генерации.

**R-07. Конкретный fallback-алгоритм при полном отказе MCP.**
Если все MCP-вызовы (включая быстрые) падают:
1. Предложить пользователю проверить `STITCH_API_KEY`
2. Предложить перезапустить сессию Claude Code
3. НЕ пытаться обходить через curl / SDK / inline-скрипты — это тупик

**R-08. Ограничение на количество retry-попыток.**
Максимум 2 retry для генерации. После 2-го неудачного retry — остановиться, сообщить пользователю, не продолжать хаотичные эксперименты.

### Пользовательский опыт — до генерации

**R-09. Минимум вопросов при первом запуске.**
Для первой генерации спрашивать только то, что нельзя вывести из контекста. Дефолты:
- Устройство: DESKTOP (если не указано иное)
- Варианты: 1 (для первого запуска)
- Creative range: не спрашивать (EXPLORE по умолчанию)
- Variant aspects: не спрашивать (все аспекты по умолчанию)

Варианты и тонкие настройки предлагать ПОСЛЕ первой генерации: "Нравится? Хочешь 3 варианта с разными лейаутами?"

**R-10. Не показывать полный промпт на английском.**
Показать пользователю краткое описание на его языке: "Отправляю: дашборд с сайдбаром, недельным календарём, sage/amber палитра..."
Полный английский промпт сохранять в `prompts.md`, но не выводить в чат (если пользователь не просит).

### Пользовательский опыт — во время генерации

**R-11. Прогресс-индикация.**
Перед генерацией сообщить: "Генерирую дизайн (~1 минута)..."
Не оставлять пользователя в тишине на время долгой операции.

### Пользовательский опыт — после генерации

**R-12. Inline-превью через Read tool.**
После скачивания PNG-превью — показать его прямо в чате через Read tool (Read поддерживает отображение изображений). Пользователь видит результат без переключения в браузер.

**R-13. Обязательная запись tracking-файлов.**
После каждой успешной генерации обновить ВСЕ tracking-файлы:
- `context-map.md` — ID проекта, экрана, пути к файлам, связи
- `prompts.md` — отправленный промпт с датой и параметрами
- `usage.json` — расход кредитов

Это должно происходить как обязательный шаг, а не опционально.

**R-14. Структурированный вывод результатов.**
После генерации показать:
- Inline-превью (R-12)
- Пути к файлам
- Краткое описание: что на экране
- Следующие шаги: "Доработать? Ещё варианты? Извлечь тему?"

### Архитектура и надёжность

**R-15. Идемпотентность операций.**
Любой шаг скилла можно безопасно повторить без побочных эффектов. В частности:
- `create_project` при наличии активного → пропустить, вернуть существующий ID
- `download` при наличии файла → перезаписать (не падать)
- Запись в context-map → upsert, не дублировать записи

**R-16. Документировать известные ограничения MCP-транспорта.**
Добавить в скилл секцию:
```
Known Issue: generate_screen_from_text may timeout via MCP (ECONNRESET after ~60s).
This is a transport-layer limitation, not a Stitch API error.
The generation usually completes server-side despite the connection reset.
Use fire-and-poll strategy (R-04).
```

**R-17. Единый формат путей в stitch-output.**
Стандартизировать структуру:
```
./stitch-output/{project-title-slug}/
  ├── {screen-title-slug}/
  │   ├── index.html
  │   └── preview.png
  ├── {screen-title-slug}-variant-1/
  └── {screen-title-slug}-edit-1/
```
Человекочитаемые имена вместо ID (`mentoring-booking/calendar/` вместо `project-608/screen-9ef3/`).

## Приоритеты

| Приоритет | Требования | Обоснование |
|-----------|-----------|-------------|
| P0 — блокер | R-04, R-05, R-08, R-16 | Без fire-and-poll скилл нерабочий для генерации |
| P0 — блокер | R-01, R-02, R-03 | 5 дублей проектов — критичная проблема |
| P1 — важно | R-09, R-10, R-11, R-12 | UX: меньше вопросов, больше визуала |
| P1 — важно | R-13, R-15 | Надёжность tracking-файлов |
| P2 — улучшение | R-06, R-07, R-14, R-17 | Полировка и удобство |
