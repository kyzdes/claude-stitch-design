# Plan: Google Stitch Design Plugin для Claude Code

## Context

Нужен Claude Code плагин для интеграции с Google Stitch API — AI-сервис генерации UI-дизайнов из текстовых промптов (HTML + Tailwind CSS + PNG). Цель: прямо из Claude Code генерировать интерфейсы, итерировать дизайны, извлекать темы и встраивать результаты в проекты.

Stitch предоставляет MCP endpoint (`https://stitch.googleapis.com/mcp`) с 14 инструментами и TypeScript SDK (`@google/stitch-sdk`). Существуют docs/PRD.md и docs/api-specs.md с полной спецификацией.

**Подход**: Hybrid — MCP для прямого доступа к 14 Stitch-инструментам + скиллы для оркестрации workflows + утилитные скрипты для файловых операций.

---

## Как это работает для пользователя

### Установка (один раз)
1. Добавить `STITCH_API_KEY` в env (получить на https://stitch.withgoogle.com/settings)
2. Установить плагин в Claude Code: `/plugin` → указать путь к папке

После этого Claude получает доступ к 14 Stitch MCP-инструментам, а скиллы учат его КАК правильно ими пользоваться.

### Основной сценарий: Conversational Design Flow

Главный workflow — это **диалоговый процесс дизайна**, а не просто "промпт → картинка":

```
┌─────────────────┐     ┌──────────────────┐     ┌──────────────────┐     ┌──────────────┐
│  1. Брейншторм  │────▶│ 2. Сбор парам-в  │────▶│ 3. Составление   │────▶│ 4. Генерация │
│  (опционально)  │     │  (стиль, device,  │     │   промпта для    │     │  + открытие  │
│                 │     │   кол-во вар-в)   │     │   Stitch         │     │  в браузере  │
└─────────────────┘     └──────────────────┘     └──────────────────┘     └──────┬───────┘
                                                                                 │
                                                                                 ▼
                                                                          ┌──────────────┐
                                                                          │ 5. Итерации  │
                                                                          │ (edit/vars)  │
                                                                          └──────────────┘
```

**Пример полного диалога:**

```
Ты: /stitch-design

Claude: Давай создадим UI-дизайн! Расскажи, что ты хочешь сделать?
        Можешь описать идею развёрнуто — я помогу превратить её в ТЗ.
        Или кратко опиши, что нужно, и перейдём к настройкам.

Ты: Хочу сделать приложение для трекинга привычек. Что-то вроде
    Streaks, но с геймификацией — бейджи, уровни, визуальный прогресс.

Claude: Интересная идея! Давай проработаем:
        - Какой основной экран? Дашборд привычек с прогрессом?
        - Геймификация: бейджи в стиле достижений или система уровней?
        - Целевая аудитория: молодёжь (яркий, playful) или универсальная?
        ...

Ты: Дашборд с карточками привычек, каждая карточка показывает streak.
    Бейджи как достижения. Яркий стиль для молодёжи.

Claude: Отлично! Собираю параметры:
        - Устройство: MOBILE (приложение для привычек)
        - Стиль: яркий, playful, с градиентами — молодёжная аудитория
        - Сколько вариантов? Рекомендую 3 для начала
        - Диапазон: EXPLORE — умеренные вариации
        Всё верно? Генерирую?

Ты: Да, давай

Claude: [генерирует 3 варианта через Stitch]
        Готово! Открываю в браузере...
        - Вариант 1: ./stitch-output/var-1/index.html ← градиентные карточки
        - Вариант 2: ./stitch-output/var-2/index.html ← неоновый стиль
        - Вариант 3: ./stitch-output/var-3/index.html ← пастельный
        Какой нравится? Хочешь доработать?

Ты: Второй крутой, но сделай иконки покрупнее и добавь bottom nav bar
Claude: [edit через Stitch] → обновлённый HTML в браузере
```

**С референсом:**
```
Ты: /stitch-design Хочу дашборд как в Linear, но для трекинга задач
    [прикрепляет скриншот Linear]

Claude: Отлично, вижу скриншот. Нравится минималистичный стиль Linear:
        - Сдержанная палитра, много whitespace
        - Компактные строки таблиц с иконками статусов
        Сохраняю референс и ТЗ... [→ stitch-design/design-requirements.md]

        Параметры:
        - Устройство: DESKTOP
        - Стиль: минималистичный à la Linear
        - Варианты: 3
        Генерирую?

Ты: Да

Claude: [загружает скриншот как референс, генерирует]
        → 3 варианта в браузере
        [→ обновляет prompts.md, context-map.md]
```

**Короткий сценарий (без брейншторма):**
```
Ты: /stitch-design Мне нужен лендинг для AI-стартапа

Claude: Окей, пару вопросов:
        - Стиль? (минималистичный / корпоративный / bold-яркий)
        - Устройство? (desktop / mobile / оба)
        - Сколько вариантов? (1-5)
        - Есть пример того, что нравится? (ссылка, картинка — необязательно)

Ты: Минималистичный, desktop, 2 варианта

Claude: [генерирует] → открывает в браузере
        [→ сохраняет промпт и карту в stitch-design/]
```

### Как открывается результат

Stitch возвращает **полный self-contained HTML** с inline Tailwind CSS, Google Fonts, Material Symbols. Скилл:
1. Скачивает HTML + PNG в `./stitch-output/`
2. Запускает `open ./stitch-output/xxx/index.html` — файл сразу открывается в браузере
3. PNG-скриншот сохраняется для превью/сравнения вариантов

### Дополнительные сценарии (после основного flow)

- **Редактирование**: "Поменяй цвет хедера" → edit через Stitch → обновлённый HTML
- **Варианты**: "Покажи ещё 3 варианта в другом стиле" → generate_variants
- **Извлечение темы**: "Дай мне Tailwind конфиг из этого дизайна" → парсинг HTML → файлы конфигурации
- **Загрузка wireframe**: "Вот скетч [файл], сделай из него UI" → upload → Screen
- **Дизайн-система**: "Примени единый стиль ко всем экранам" → design system

### Что пользователь НЕ делает
- Не пишет код и не вызывает API напрямую
- Не устанавливает npm-зависимости (скрипты zero-dep)
- Не настраивает MCP вручную (плагин делает это автоматически)
- Не составляет промпты для Stitch вручную — скилл делает это из контекста диалога

**Изменение от PRD**: вместо 7 скиллов — 4. Главный скилл `stitch-design` объединяет брейншторм + генерацию + варианты + дизайн-системы в один conversational flow. Отдельные model-invoked скиллы только для действий, которые вызываются в уже существующем контексте (edit, theme, upload).

## Фаза 1: Инфраструктура + главный скилл

### 1.1 `.claude-plugin/plugin.json`
```json
{
  "name": "stitch-design",
  "description": "Google Stitch AI UI generation — brainstorm, generate, edit, iterate on UI designs from text prompts",
  "version": "0.1.0"
}
```

### 1.2 `.mcp.json` — Remote HTTP с headers
```json
{
  "stitch": {
    "type": "http",
    "url": "https://stitch.googleapis.com/mcp",
    "headers": {
      "x-api-key": "${STITCH_API_KEY}"
    }
  }
}
```
Паттерн подтверждён GitHub/Greptile плагинами. `${STITCH_API_KEY}` подставляется из env.

### 1.3 `scripts/download.mjs` — Zero-dep файловый загрузчик
- Принимает URL + output path через CLI args
- Скачивает через `fetch()`, сохраняет через `fs`
- Создаёт директории через `mkdirSync({ recursive: true })`
- Использование: `node scripts/download.mjs <url> <output-path>`

### 1.4 `skills/stitch-design/SKILL.md` — ГЛАВНЫЙ скилл (Conversational Design Flow)

User-invoked: `/stitch-design [промпт]`. Также model-invoked при упоминании UI/дизайна/прототипа.

**Скилл инструктирует Claude следовать conversational flow из 5 шагов:**

**Шаг 0: Инициализация**
- Создать папку `./stitch-design/` в проекте (если не существует)
- Эта папка — рабочее пространство дизайна, хранит все промежуточные файлы

**Шаг 1: Брейншторм + референсы (опциональный)**
- Если пользователь дал расплывчатое описание → помочь проработать идею:
  - "Какая цель экрана? Кто целевая аудитория?"
  - "Какие ключевые элементы? Что вдохновляет?"
  - Итерация до чёткого ТЗ
- **Референсы** — пользователь может приложить:
  - **Ссылку на сайт** → Claude анализирует (через MCP chrome tools или описывает)
  - **Картинку** (скриншот, wireframe) → можно отправить в Stitch через `upload_screens_from_images`
  - **HTML-код** (например из предыдущего варианта) → Claude анализирует стиль
  - **Текстовое описание стиля** ("как Linear", "в стиле Notion")
- Если пользователь дал конкретное описание → пропустить, перейти к шагу 2
- Если пользователь явно говорит "без брейншторма" → пропустить
- **Результат** → сохранить в `./stitch-design/design-requirements.md`

**Шаг 2: Сбор параметров (через AskUserQuestion)**
- Стиль: минималистичный / корпоративный / bold-яркий / playful / etc.
- Устройство: DESKTOP / MOBILE / TABLET
- Количество вариантов: 1-5 (рекомендация: 3)
- Creative range: REFINE / EXPLORE / REIMAGINE (объяснить разницу)
- Референсы (если ещё не предоставлены): "Есть пример того, что нравится?"

**Шаг 3: Составление промпта**
- Claude компилирует из контекста диалога + референсов оптимальный промпт для Stitch
- Показывает промпт пользователю для одобрения
- Пользователь может скорректировать
- **Результат** → сохранить в `./stitch-design/prompts.md` (все промпты, включая итерации)

**Шаг 4: Генерация + показ**
- **Перед генерацией** — показать текущий расход кредитов (из локального трекера)
- `create_project` → `generate_screen_from_text` (или `generate_variants`)
- Если есть референс-картинка → `upload_screens_from_images` + `edit_screens`
- Скачать HTML/PNG через `scripts/download.mjs`
- Открыть в браузере: `open ./stitch-output/.../index.html`
- Показать пути к файлам
- **Обновить** `./stitch-design/context-map.md` — карта проекта:
  - Какие экраны сгенерированы, их ID и пути
  - Какие референсы использовались
  - Связи между экранами (оригинал → edit → variant)
- **Обновить** `./stitch-design/usage.json` — счётчик кредитов

**Шаг 5: Итерация**
- "Хочешь доработать? Что изменить?"
- Можно приложить новый референс для корректировки
- При ответе → `edit_screens` или `generate_variants`
- Каждая итерация сохраняется + обновляется `context-map.md` и `prompts.md`

### Рабочая папка `./stitch-design/`

```
./stitch-design/
├── design-requirements.md    # ТЗ из брейншторма: цель, аудитория, ключевые элементы
├── prompts.md                # Все промпты отправленные в Stitch (с итерациями)
├── context-map.md            # Карта проекта: экраны, их ID, связи, референсы
├── usage.json                # Трекер кредитов (дата, тип операции, расход)
└── references/               # Скопированные/скачанные референсы
    ├── screenshot-1.png
    └── notes.md              # Описания ссылок-референсов
```

Эти файлы позволяют:
- **Воспроизвести** дизайн-процесс в новой сессии Claude Code
- **Продолжить** работу над проектом позже — Claude читает context-map и понимает что уже сделано
- **Поделиться** контекстом с коллегами

### Трекинг кредитов

**Лимиты (Google AI Pro):**
- 400 ежедневных бонусов / день (генерации, edit, variants)
- 15 Daily Redesign Credits / день

**Проблема**: Stitch API (14 MCP tools) не предоставляет эндпоинт для проверки квоты.

**Решение — локальный трекинг** через `./stitch-design/usage.json`:
```json
{
  "daily": {
    "2026-03-19": {
      "generations": 5,
      "edits": 2,
      "variants": 9,
      "uploads": 1,
      "total": 17
    }
  },
  "limits": {
    "daily_bonuses": 400,
    "daily_redesign": 15
  }
}
```

**Скилл перед каждой генерацией**:
1. Читает `usage.json`, проверяет дату (если новый день — сбрасывает счётчик)
2. Показывает: `"Использовано: 17/400 кредитов сегодня. Генерирую 3 варианта (расход ~3). Продолжить?"`
3. После генерации обновляет счётчик
4. При приближении к лимиту (>350) — предупреждает жёлтым
5. При достижении лимита — блокирует с сообщением

## Ключевые решения

| Решение | Обоснование |
|---------|-------------|
| Remote HTTP MCP (не StitchProxy) | Claude Code поддерживает `headers` — проверено по GitHub/Greptile |
| Скиллы вместо CLI | Claude Code сам является оркестратором; отдельный CLI избыточен |
| Zero-dep скрипты для download/parse | Минимум зависимостей для базовой работы |
| 1 скилл = 1 workflow | Claude загружает только релевантный скилл, экономя контекст |
